The hard questions, answered

“But what about…”

“Agents write sloppy code — who's accountable when it breaks?”
You are. Accountability doesn't move. The agent is your team; a good EM owns their team's output. Your standards, specs, and review are how you enforce quality.
“Won't I just spend all day reviewing bad PRs?”
Only if your context is bad. Better AGENTS.md, specs, and CI gates mean better first drafts. Reviewing is leverage — one review, many PRs — not busywork.
“If a cloud agent ships a bug, is that on me?”
Yes — and that's the point. You own the service. The repro-test-first bug flow and the traceability gate exist so the machine catches most of it before you do.
“This is more management, less real engineering.”
It's more leverage, not more meetings. Architecture, system design, and directing agents is the real engineering now.

Shift 01 · A hat you wear

Technical Product Manager

Facing up — toward the product team. You stop being an order-taker and start co-owning the product.

What it asks of you

  • → Talk to the product team directly — understand the user and the business, don't just consume a ticket.
  • → Treat the Jira ticket as a PRD — the what and why. The technical spec is your deliverable.
  • → Push back, ask the product questions, and shape scope before anything gets built.
  • → Offer more than code: prioritization, trade-offs, and a point of view on where the service should go.

The mindset to overcome

The "I'm only a programmer, someone else decides the what" reflex. From here, you are one of the people who decides.

Shift 01 · The new role

Engineering Manager

The umbrella role. Your agents are your team; you are their lead — and you wear whatever hats the work needs beneath it.

What it asks of you

  • → Coordinate the whole effort — break work down, delegate to agents, sequence it.
  • → Give your team what a good manager gives: clear context, clear specs, clear standards.
  • → Review their output critically and own it — their PRs are your PRs.
  • → Invest in making the team better: sharper prompts, better harness, tighter feedback loops.

The mental model

You wouldn't hand-write everything if you managed five engineers. You'd lead them. Same here — the team just happens to be agents.

Shift 02 · Ownership

What owning a service means

For a period — roughly three months — one service is yours, end to end.

You own

  • → The roadmap — what gets built, and in what order.
  • → The architecture and the quality bar.
  • → The outcomes — what ships and how it behaves in production.
  • → The agents doing the work — and the context that makes them effective.

Why a whole service, not tickets

Depth compounds. Context, consistent decisions, and momentum only build when one person owns the whole picture over time.

Shift 03 · Standards · agentic-standards-kit

The agentic harness, exactly

Drop-in and versioned. Apply it to any service repo with bin/sync.ps1 / bin/sync.sh and the repo carries its own standards and CI gates. Dispatch & orchestration are centralized — see below.

agentic-standards-kit/ ├── AGENTS.md ← rules every AI agent reads first (<80 lines) ├── CONTEXT.md ← domain glossary — how you name things ├── docs/ │ ├── adr/ ← 0001 methodology · 0002 verification · 0003 autonomy tiers │ ├── agentic-dispatch.md ← how dispatch works (via radix-agent-control) │ ├── agent-ready-checklist.md ← repo readiness audit │ └── metrics.md ← KPIs + autonomy promotion bar ├── specs/_template/spec.md ← EARS criteria with stable AC IDs ├── .claude/ ← Claude Code hooks + /sync-jira skill ├── .cursor/ ← Cursor hooks, Atlassian MCP, cloud-agent env ├── .github/workflows/ ← CI gates only — dispatch lives in radix-agent-control │ ├── spec-traceability.yml ← deterministic definition-of-done gate │ └── label-prs.yml ← fact-derived auto-labels (spec / impl) └── bin/sync.ps1 · bin/sync.sh ← apply / update the kit in a repo

The four you touch most

  • AGENTS How to work a ticket: Story → EARS spec first; Bug → failing repro test named with the key; Task → trivial path or ADR. Spec merges are the human approval gate.
  • CONTEXT The glossary agents read before naming anything. Missing a term? Add it before you continue.
  • ADR Architecture Decision Records — the durable why behind every big call.
  • SPECS EARS acceptance criteria with stable AC IDs; tests reference the IDs verbatim, and the traceability gate enforces coverage.

Dispatch lives centrally — radix-agent-control

The dispatch and orchestration workflows no longer sit in every repo. They live once in RadixInc/radix-agent-control. From there a single cloud agent can span several repos in one ticket and open a repo-scoped PR in each — no per-repo duplication to maintain.

Shift 04 · Automation · The cloud flow

The cloud flow — most of the work

Bugs and spec'd stories alike. You run the whole thing from Radix Flows and dispatch cloud agents — you rarely open a local checkout.

How it flows

  • 1 A ticket lands in Radix Flows — where you manage the entire flow.
  • 2 Needs a spec? You review and edit the spec right there, then merge it — that merge is the approval gate.
  • 3 You dispatch a cloud agent through radix-agent-control (the central control plane). One agent can span several repos in the ticket and open a repo-scoped PR in each. For bugs, a failing repro test named with the key comes first.
  • 4 You review and merge. CI and the traceability gate have already run.

Why this is the default

Most tickets — even ones with a spec — never need you in an editor. Radix Flows plus cloud agents carry them, so your attention is free for the work that actually needs judgment.

Shift 04 · Automation · Hands-on

Hands-on — the complex few

The minority of tickets that need real decisions or architecture. Here you work directly with a local agent.

How it flows

  • 1 A genuinely complex change — new capability, cross-cutting design, real trade-offs.
  • 2 You work with a local agent (Claude Code, Cursor) in your checkout, spec-first, making the calls as you go.
  • 3 You open the PR and own the outcome. This is where your engineering judgment earns its keep.

When to choose it

Only when the decision density is high enough that steering an agent live beats dispatching one. The default is always the cloud flow.

Playbook · Set up once · per person

Your machine & tools

Fifteen minutes, once. You need all four even if you dispatch everything from the cloud — half the value is the local agent that shapes specs and grills your plans.

Install

  • ghGitHub CLI — winget install GitHub.cli (or brew install gh), then gh auth login. How you review & merge PRs from the terminal; both local agents shell out to it.
  • claudeClaude Code — npm i -g @anthropic-ai/claude-code, then claude in any repo. Your local agent; reads AGENTS.md/CONTEXT.md; runs /sync-jira, /grill-me.
  • cursorCursor CLI — the cloud-agent driver; sign in so its GitHub integration is wired. The launch key (CURSOR_API_KEY) lives only in the control plane — you never handle it.
  • mcpAtlassian MCP — connect the Rovo MCP (mcp.atlassian.com/v1/mcp/authv2) so the agent reads Jira/Confluence and comments back. Powers /sync-jira.

Done when

gh auth status is green, claude launches and shows skills, Cursor is signed in, and your agent can read a Jira ticket.

Playbook · Set up once · per repo

Onboard a repo

A repo has to earn cloud-agent routing — the payoff is real, but only if the agent can actually run the code. In order:

The steps

  • 1Sync the kit — bin/sync.ps1 / bin/sync.sh from inside the repo. Copies the gates, templates and config; writes .kit-version. Files you've customised get a .kit sibling to diff-and-merge by hand.
  • 2Fill AGENTS.md — mission, stack, the real build/test commands. Under 80 lines. Let the local agent draft it, then review every line — this file governs everything.
  • 3Bootstrap CONTEXT.md — run /grill-with-docs against the codebase; approve 20–40 real domain terms. Grow it inline, never batch.
  • 4Fill .cursor/environment.json — real install/start/test commands, ports, secrets. This is what lets an agent stand the app up and verify.
  • 5Prerequisites — the DISPATCH_APP GitHub App (Contents + PRs read/write), the Cursor integration, secrets DISPATCH_APP_ID/_PRIVATE_KEY + ANTHROPIC_API_KEY, and the agentic-dispatch topic. CURSOR_API_KEY lives only in the control plane.

The one that bites

Flaky tests. An agent runs the suite 20+ times per ticket — every flake is 20× more expensive. Quarantine or fix flakiness before enabling agents. Non-negotiable.

Playbook · Your repo after sync

Touch / don't touch

Hand-editing the machinery is how the guarantees quietly stop holding. Know which is which.

Yours to own

  • editAGENTS.md — the rules every agent reads. CLAUDE.md is a one-line shim that imports it.
  • editCONTEXT.md — the domain glossary; grow it via /grill-with-docs.
  • edit.cursor/environment.json — the cloud-agent run recipe.

Generated per ticket — you review & merge

  • autospecs/<KEY>/spec.md — written by the triage agent (or you via /sync-jira).

Don't touch — the machinery

  • lockspec-traceability.yml + check-spec-coverage.mjs — the deterministic definition-of-done gate.
  • lockping-implement-on-spec-merge.yml — pings the control plane on spec merge (org-portable, keys on the PR title marker).
  • locklabel-prs.yml + labeler.yml — fact-derived labels. pull_request_template.md — the AC + acceptance-evidence template.
Playbook · A ticket, start → finish

From groomed to Done

Two triggers in, two human merges, everything between automated.

The loop

  • 1Groom to Definition of Ready (PM, in Jira) — Story: EARS acceptance criteria; Bug: repro + expected vs actual; Task: the constraint. No terms missing from CONTEXT.md.
  • 2Dispatch — from the Jira Dispatch Agent panel or the Radix Flows app. Both fire the same event at radix-agent-control; pick repos, spec_mode, optional instructions.
  • 3Triage agent writes specs/<KEY>/spec.md and opens a <KEY> SPEC: … PR (one per repo).
  • 4You review & MERGE the spec PR — approval gate #1 (the developer, not the PM). The merge is the green light.
  • 5Completeness check passes → the implementation agent works test-first, boots the real app, and opens a <KEY>: … PR with a ## Acceptance evidence section.
  • 6You review the evidence & MERGE — approval gate #2.
  • 7Jira auto-transitions the ticket to Done. No manual status update.

The one rule to remember

The PR title is the only signal automation trusts — <KEY> SPEC: green-lights implementation, <KEY>: closes the ticket. Branch names carry nothing (cloud agents live on cursor/*).

Playbook · Specs & commands

Specs, the grill workflow & the cheat-sheet

For one ticket the agent drafts the spec; for a project you shape it, then let the agent fill it in. Same tools either way.

The authoring loop

  • →/grill-with-docs — align the plan to the domain model; update CONTEXT.md / ADRs inline as decisions crystallise.
  • →/sync-jira <KEY> — pull the ticket, draft specs/<KEY>/spec.md in EARS, comment the AC list back to Jira.
  • →/grill-me — get interrogated until every branch resolves. This is how you write and modify specs.
  • →/to-issues · /tdd — slice a project into tickets; red-green-refactor, one failing test per AC.

EARS acceptance criteria

Each AC is one testable statement with a stable ID — AC-1: When <event>, the system shall <response>. IDs never renumber once approved; tests reference them verbatim, and the traceability gate enforces coverage.

Playbook · When you're stuck

The FAQ

The questions people actually ask.

  • QDo I merge the agent's PR, or does it self-merge? You merge — always. Two human merges (spec, then impl); agents never merge. Everything between is automated.
  • QNothing happened after I merged the spec. Usually a multi-repo ticket — implementation only starts once the spec is on the default branch of every repo. Otherwise check the ping-implement run in your repo and implement-orchestrate in the control plane.
  • QThe spec looks wrong — can I skip it and edit code? No. Fix the spec (re-grill, edit specs/<KEY>/spec.md), then merge. The gate blocks the impl PR without it.
  • QCI is stuck on a check that never reports. A required check got skipped by a path filter — it must “always run and self-skip” like spec-traceability. Repo-owner fix; don't disable branch protection.
  • QCursor: “failed to verify existence of branch.” The control-plane CURSOR_API_KEY isn't a User/Service key, or lacks access to the target repo. Flag the control-plane owner.
  • QCan I put a screenshot in the PR to prove it works? No — no evidence files in git, anywhere. The acceptance account goes in the PR body; visuals stay in the linked agent run.
