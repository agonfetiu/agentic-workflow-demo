?"I'm confused" / FAQ

You just watched the demo. This is how you actually do it.

The presentation covered the why — the shift from writing every line yourself to directing agents that do. This playbook is the how: exactly what to install, what to touch (and what never to touch), and what happens to a ticket from the moment it's groomed to the moment it's Done.

The whole flow in one line:
Dispatch (Jira panel or Radix Flows) → agent writes spec PR → you review + merge the spec → agent writes implementation PR with acceptance evidence → you review + merge → ticket Done.

Read it top-to-bottom once, then keep it open as a reference. Every step below expands — click any card to see who does it, what you'll see on screen, what you're expected to do, and the gotchas that bite people.

Jira (where work starts), GitHub (where orchestration lives), Cursor (the cloud agent), Claude Code (your local agent).

A spec (the agreed behaviour) and an implementation PR (the code + acceptance evidence). Each is a separate PR you merge.

You approve the spec by merging it. You approve the code by merging it. Everything between is automated.

🤖1 control plane

radix-agent-control launches the agents. Your repo just carries CI gates + one small ping workflow.

Orientation

The mental model read first

Five ideas. Once these click, every screen in the demo makes sense.

◆The ticket is a PRD, not a design›

A Jira ticket tells you what users need and why. It is not the technical design — even when it reads as detailed. The design is a separate, machine-checkable deliverable called the spec, and it must be merged before any code is written. That separation is the whole point: it's where a human still says "yes, build this."

◆The PR title is the only signal automation trusts›

Every piece of automation keys on the PR title, never the branch name. Cloud agents work on Cursor-designated cursor/* branches, so branches carry no meaning.

  • SPEC  EN-1234 SPEC: … — a spec PR. Merging it green-lights implementation.
  • IMPL  EN-1234: … — an implementation PR. Merging it closes the ticket.
ℹ️
The exact marker regex is ^[A-Z][A-Z0-9]*-\d+ SPEC:. If you rename a PR title and drop the marker, the automation stops seeing it. Don't hand-edit agent PR titles.
◆GitHub owns orchestration — a control plane launches the agents›

One repo, RadixInc/radix-agent-control, is the control plane. It holds no product code — its only job is to launch Cursor cloud agents into other repos and follow them to completion. A single Jira ticket can span several repos (a backend + a frontend); one agent takes all of them in one context window so the cross-repo contract stays coherent, then opens a repo-scoped PR in each.

Your repo does not carry the launch logic or the Cursor secret. It carries the CI gates and one tiny "ping" workflow that tells the control plane "my spec merged."

◆The definition of done is enforced in layers›

Three layers, in order (full detail in The verification stack):

  • The build proves existence — a script (spec-traceability) blocks any PR missing its type-appropriate artifact. Not a judgment call.
  • The agent proves behaviour — it boots the real app in its sandbox and exercises the change end-to-end before opening the PR.
  • The human confirms on evidence — you verify the per-AC acceptance account in the PR body instead of reconstructing it.
◆Everything is stateless and idempotent›

No cross-repo state is held anywhere. When a spec merges, the repo pings the control plane, which re-reads live GitHub truth: "is the spec on the default branch of every repo in this group?" The last spec merge to land is the one that trips implementation. That means retries, re-runs, and out-of-order merges are all safe.

Step-by-step — the exact events & files

Every arrow above, spelled out. Expand any phase.

APhase A · Dispatch  two triggers → one control plane›
  1. 1 · Human → Jira or Radix Flows. Groom the ticket, then dispatch from either trigger:
    • Jira — the Dispatch Agent panel (Forge app RadixInc/forge-dispatch-agent): pick repos[] (from the agentic-dispatch topic), a spec_mode, optional instructions.
    • Radix Flows — the same repo multi-select + mode, from the Flows board.
  2. 2 · trigger → control plane. Whichever you used fires the same repository_dispatch type: dispatch-agent at <org>/radix-agent-control with {ticket_key, repos, spec_mode, instructions?, dispatched_by}. (Both mint a token from the DISPATCH_APP — no PAT.) From here the two paths are identical.
  3. 3 · Control plane (internal). triage-orchestrate.yml validates the ticket key + mode, dedups, and resolves repos — using them verbatim if supplied, or auto-routing from catalog.json (a Claude Haiku classifier) when omitted.
  4. 4 · Control plane → Cursor. _launch-cursor-agent.yml (phase=triage) calls POST api.cursor.com/v1/agents with the repos[] array and autoCreatePR:true (HTTP Basic, a User/Service CURSOR_API_KEY), then follows the run.
BPhase B · Spec  agent writes → human merges›
  1. 5 · Cursor → repo(s). One agent, all repos in context. In each it reads AGENTS.md + CONTEXT.md, decides the phase, and (for a Story) writes specs/<KEY>/spec.md on its cursor/* branch.
  2. 6 · Cursor → repo(s). Opens a PR titled <KEY> SPEC: … (ready for review) with a > Repos: header listing the group — one PR per repo.
  3. 7 · Repo → Jira. GitHub-for-Jira links the PR by the key in its title; Automation Rule D comments "review & merge to green-light implementation."
  4. 8 · Human → repo(s). Review each spec PR and merge — the primary developer, not the PM. approval gate #1
CPhase C · Implement  ping → completeness → agent›
  1. 9 · Repo (internal). The spec merge to the default branch fires ping-implement-on-spec-merge.yml — it keys on the <KEY> SPEC: title marker (loop-safe: impl PRs lack it), reads the group from the spec's > Repos: header, and derives the same-org control repo from $GITHUB_REPOSITORY_OWNER.
  2. 10 · Repo → control plane. Fires repository_dispatch type: implement-ticket with {ticket_key, repos}. (Token minted from the DISPATCH_APP GitHub App — no PAT.)
  3. 11 · Control plane (internal). implement-orchestrate.yml runs the completeness check: is specs/<KEY>/spec.md on the default branch of every repo in the group? Reads live GitHub state. No → no-op (wait for the next ping). Yes → launch.
  4. 12 · Control plane → Cursor. _launch-cursor-agent.yml (phase=implement) launches one agent across the whole group.
  5. 13 · Cursor (internal). TDD: one failing test per AC (tagged AC-<n>:) → implement → boot the real app and acceptance-test the change end-to-end.
  6. 14 · Cursor → repo(s). Opens a PR titled <KEY>: … with a ## Acceptance evidence section (per-AC table + run output). No evidence files in the diff.
  7. 15 · Repo (internal). Per-repo CI runs: spec-traceability (the deterministic gate), label-prs, and AI review.
  8. 16 · Repo → Jira. Automation Rule E comments the PR link and transitions the ticket → Code Review.
DPhase D · Close  human merges → Jira done›
  1. 17 · Human → repo(s). Review the acceptance evidence and merge the impl PR. approval gate #2
  2. 18 · Repo → Jira. The merged PR title has no SPEC marker, so Automation Rule C transitions the ticket → Done. No manual status update.
🔑The title is the API

Automation keys on the PR title marker (<KEY> SPEC: / <KEY>:), never the branch — cloud agents live on cursor/* branches.

🧭Stateless by design

Nothing holds cross-repo state. The completeness check reads live GitHub truth on every ping, so the last spec merge in a group is the one that trips implement.

🔁Org-portable, no PAT

The ping derives its control repo from $GITHUB_REPOSITORY_OWNER and mints a token from the DISPATCH_APP. Same file works in any org.

🧩One agent, N repos

A multi-repo ticket = one Cursor agent holding every repo in one context → a coherent contract → one repo-scoped PR per repo.

Set up once · per repository

2 · Onboarding a repo once per repo

This is the repo owner's job (with the lead's sign-off). A repo has to earn cloud-agent routing — the payoff is real, but only if the agent can actually run the code. Follow these in order.

1Sync the Agentic Standards Kit into the repo›

The kit is a sync source, not a GitHub template. From inside the target repo, clone the kit next to it and run the sync script. It copies the workflows, PR template, spec template, and config in, and writes a .kit-version recording which version you sourced.

Windows / PowerShellUnix / CI
from inside your repo
git clone https://github.com/RadixInc/agentic-standards-kit ../agentic-standards-kit
pwsh -File ../agentic-standards-kit/bin/sync.ps1
from inside your repo
../agentic-standards-kit/bin/sync.sh
SituationWhat sync does
File missing in your repoCopies it in.
A companion file exists (AGENTS.md, CONTEXT.md, .claude/settings.json, .cursor/*)Writes a .kit sibling; leaves yours intact so you diff-and-merge by hand.
A non-companion file existsSkips it (use -Force / FORCE=1 to overwrite).
Kit-internal file (README.md, bin/, KIT_VERSION)Never copied.
ℹ️
Updating later: when a new kit version ships, re-run sync. New files copy outright; anything you customised gets a fresh .kit companion to diff against.
2Fill in AGENTS.md (the agent's constitution)›

AGENTS.md is read by every agent that touches the repo (Cursor, Claude Code, Codex, Copilot). CLAUDE.md is just a one-line shim that imports it. Keep it under 80 lines. You only need to fill three things — the rest ships filled-in and correct:

  • Mission — one paragraph: what this service does for the business.
  • Stack — one line, e.g. TypeScript / Node 20 / Express / MongoDB / Vitest / Playwright.
  • Build, test, lint — the real install / dev / test / lint / typecheck commands.
✓
Let the local agent do the first draft. From inside the repo run claude and say: "Read the codebase and fill in the Mission, Stack, and Build/test/lint sections of AGENTS.md. Keep it under 80 lines." Then review every line — this file governs everything.
3Bootstrap CONTEXT.md with /grill-with-docs›

CONTEXT.md is the domain glossary — the shared vocabulary the agent uses to pick the right names and avoid the wrong ones. Don't write it by hand. Run the skill against the existing codebase and it proposes the top domain terms; you approve the real ones and reject noise.

inside claude, from the repo
/grill-with-docs

Aim for 20–40 entries on the first pass. Each entry is one term, one-sentence definition, optional one-sentence constraint. Keep an anti-glossary too — terms you deliberately don't use (e.g. "Survey edit" → we say "new Survey version").

ℹ️
This is the "local agent that builds docs from your codebase + context" the presentation referred to. Update it inline during future grill sessions — never batch. When an agent hits a term not in the glossary, it's supposed to stop and add it via /grill-with-docs before continuing.
4Fill in .cursor/environment.json (so the agent can run the code)›

This is the single most important file for cloud agents. An agent iterates by running tests 20+ times per ticket — if it can't stand the app up, it can't verify anything. Put the real install / start / test commands, the ports, the secrets list, and the network allowlist.

.cursor/environment.json
{
  "name": "my-service cloud agent env",
  "install": "pnpm install --frozen-lockfile",
  "start": "pnpm dev",
  "terminals": [
    { "name": "tests", "command": "pnpm test --watch" },
    { "name": "typecheck", "command": "pnpm typecheck --watch" }
  ],
  "secrets": ["DATABASE_URL", "ATLASSIAN_API_TOKEN", "ANTHROPIC_API_KEY"],
  "ports": [3000],
  "network": { "allowlist": ["registry.npmjs.org", "github.com", "mcp.atlassian.com"] }
}
5Wire the prerequisites for dispatch›

These connect the repo to the control plane. Mostly org-wide, so on repo #2+ they're already done.

  • GitHub App (DISPATCH_APP) installed on the repo (org-wide is fine) with Contents: read & write + Pull requests: read & write. The ping mints a token from it to reach the control plane.
  • Cursor GitHub integration installed on the repo (org-wide is fine) — this is how the agent opens its PRs.
  • Repo secrets: DISPATCH_APP_ID + DISPATCH_APP_PRIVATE_KEY (org secrets recommended). Also ANTHROPIC_API_KEY for AI review.
    note CURSOR_API_KEY is NOT needed here — it lives only in the control plane.
  • Topic: tag the repo with agentic-dispatch so it appears in both dispatch pickers (Jira panel and Radix Flows).
⚠️
The control plane (radix-agent-control) is set up once per org and holds the Cursor key + dispatch app. Cross-org targets (e.g. a different GitHub org) need their own control plane in that org.
6Wire the dispatch triggers & Jira automation rules›

Two triggers, one entry point. The RadixInc/forge-dispatch-agent Forge app adds a Dispatch Agent panel to every Jira issue, and the Radix Flows app offers the same dispatch from its board. Both fire the identical repository_dispatch: dispatch-agent at <org>/radix-agent-control — set up either or both.

On the automation side, three Jira rules keep the ticket status in sync — set once at the project level:

RuleTriggerDoes
C — close the loopPR merged, title is not a SPECTransition ticket → Done
D — spec readyPR created, title is a SPECComment "review & merge to green-light implementation"
E — impl readyPR created, title is not a SPECComment + transition → Code Review
ℹ️
These rules are notification/tracking only. CI is the hard stop — a mis-dispatched or disobedient agent run simply can't merge. Full setup in the kit's docs/jira-setup.md.
✓Final gate: the agent-ready checklist›

Before the repo goes into rotation, it must pass a yes/no audit (not a vibe). The headline items:

  • One-command hermetic setup — fresh clone → running app + passing tests, one documented command.
  • Seeded fixtures, no manual secrets for tests.
  • Suite under ~10 minutes and zero known-flaky tests (20× more expensive in this world — quarantine or fix first).
  • Runnable headless (Playwright headless, no GUI-only steps).
  • spec-traceability + the test workflow are required checks on protected branches; main is protected (no force-push, ≥1 human review).
⛔
Flaky tests are the #1 killer. An agent that runs a flaky suite 20 times will get 20 different answers and burn a whole run chasing ghosts. Fix flakiness before enabling agents — this is the one item you cannot skip.

After the sync, you'll see new files. Here's the honest map of what you edit, what's generated for you, and what you must never touch (the moment you hand-edit the gates, the guarantees stop holding).

FileWhat it is & what to do

DON'T TOUCH  The machinery — hand-editing these breaks the guarantees

FileWhat it does & why to leave it
.github/workflows/spec-traceability.ymlThe gate. Blocks any PR missing its type-appropriate artifact. Runs a deterministic script — no LLM judgment. This is "the build proves existence."
.github/scripts/check-spec-coverage.mjsThe script the gate runs (reads the ticket key from the PR title, checks every AC ID appears in a tagged test). Don't edit — it's the same across all repos.
.github/workflows/ping-implement-on-spec-merge.ymlWhen a <KEY> SPEC: PR merges to the default branch, this pings the same-org control plane to launch implementation. Org-portable by design — no per-repo edit.
.github/workflows/label-prs.yml + labeler.ymlAuto-applies fact-derived labels (spec, spec-amendment). You never label these by hand.
.github/pull_request_template.mdThe PR template: AC checkboxes + the ## Acceptance evidence section. Agents fill it; you read it.
specs/_template/spec.mdThe EARS spec template new specs are cloned from. Leave as-is.
.claude/, .cursor/hooks*Skill-routing hooks and settings. Config, not content — leave unless you know exactly what you're changing.
⚠️
CI hygiene one-liner: spec PRs are markdown-only. Give your expensive workflows (build, deploy, E2E) paths-ignore: ['specs/**','docs/**','**/*.md'] so a spec PR never triggers them. But a required check skipped by a path filter never reports and hangs the PR on "Expected" — so required checks must "always run and self-skip" (as spec-traceability already does). This is a repo-owner concern, not a per-ticket one.

4 · A ticket, start → finish the main event

This is the flow you saw in the demo, broken into every step. Expand each one for exactly who acts, what appears on screen, and what you do.

  Jira: groom → Dispatch Agent          (human triggers)
      │  repository_dispatch → radix-agent-control
      ▼
  triage agent  → writes specs/<KEY>/spec.md, opens  "<KEY> SPEC: …"  PR in each repo
      ▼
  human: review + MERGE the spec PR         ← approval gate #1
      │  repo pings control plane on merge → completeness check (live GitHub state)
      ▼
  implementation agent → test-first, boots the app, opens  "<KEY>: …"  PR w/ acceptance evidence
      ▼
  human: review evidence + MERGE            ← approval gate #2
      ▼

1Groom the ticket to "Definition of Ready"›
Who
PM / whoever grooms.
Where
Jira.

A ticket may only be dispatched once it's ready. The bar depends on type:

  • Story — EARS-testable acceptance criteria present; out-of-scope stated.
  • Bug — reproduction steps + expected vs actual + environment.
  • Task — the constraint stated ("no behaviour change") or the target state for config/infra work.
  • No domain terms missing from the repo's CONTEXT.md.
✓
The PM authors acceptance criteria on the Jira ticket. The spec mirrors them — Jira stays the source of truth for what, the repo owns how.
2Dispatch the agent — from Jira or Radix Flows›
Who
Anyone on the ticket.
Where
The Dispatch Agent panel in Jira, or the Radix Flows app.

You have two triggers — pick whichever you're already in. They fire the identical event at the control plane, so the outcome is the same:

  • Jira → Dispatch Agent panel — on the issue, added by the forge-dispatch-agent Forge app. Best when you're triaging in Jira.
  • Radix Flows app — the board that also visualises every ticket's stage/PR state. Best when you're already watching the pipeline. (Flows is a helper/visualiser — never on the critical path.)

Either way you pick:

  • Repo(s) — multi-select, discovered by the agentic-dispatch topic. Pick every repo the ticket touches; one agent spans them all.
  • spec_mode — auto (agent decides from ticket type), force-spec, or no-spec.
  • Instructions (optional) — free-text guidance with any context you have. It's guidance only — it can't override the process.

Click dispatch. The trigger fires one repository_dispatch at radix-agent-control, and GitHub runs the rest.

ℹ️
Neither handy? Test from the terminal — dispatch the control plane directly:
gh
gh workflow run triage-orchestrate.yml -R RadixInc/radix-agent-control \
  -f ticket_key=EN-1234 -f repos=RadixInc/radix-platform -f spec_mode=auto
3The triage agent writes the spec & opens the spec PR›
Who
Cursor cloud agent (automated).
You
Wait & watch the run.

The control plane's triage-orchestrate.yml launches one agent across all selected repos. In each repo it decides the phase from the ticket type (per AGENTS.md):

  • Story → writes specs/<KEY>/spec.md from the template with Status: approved, opens a PR titled <KEY> SPEC: … carrying a > Repos: header listing the group, posts product questions to Jira, then stops.
  • Bug → no spec; a failing repro test named with the key is the spec.
  • Trivial Task → claims the no-spec label.
ℹ️
Jira automation rule D comments on the ticket: "📝 Spec PR ready — review it, then MERGE to green-light implementation."
4You review & merge the spec PR  approval gate #1›
Who
The primary developer (spec is an engineering artifact — don't route it to the PM).
Where
GitHub PR.

Read the spec. Does it capture the ticket? Are the ACs testable as written? Is the negative space ("Out of scope") spec'd? If the agent left Open questions, resolve them (product questions → comment on Jira; engineering → in the spec).

Merging the spec PR is the approval. That's the whole gate — no separate sign-off ceremony. The merge fires the repo's ping-implement-on-spec-merge.yml.

⚠️
If the ticket spans multiple repos, review and merge each repo's spec PR. Implementation only starts once the spec is on the default branch of every repo in the group — the completeness check reads live GitHub state, so the last merge trips it. Order doesn't matter.
5The implementation agent builds it & opens the impl PR›
Who
Cursor cloud agent (automated).
You
Wait & watch.

The control plane's implement-orchestrate.yml confirms group completeness, then launches one implementation agent across the group. Each agent:

  • Works test-first — one failing test per acceptance criterion, tagged with its AC ID (test('AC-3: …')), before any implementation.
  • Boots the real app in its sandbox and exercises the change end-to-end. Unit tests aren't enough.
  • Opens a PR titled <KEY>: … with a ## Acceptance evidence section: a per-AC table (AC ID, how it was verified in the running app, result) + key run output.
⛔
No evidence files enter GitHub — ever. No screenshots, no acceptance.md, no evidence branches. The PR diff is code + tests only; visuals stay in the linked agent run. Rule E moves the Jira ticket to Code Review.
6You review the evidence & merge  approval gate #2›
Who
Reviewer.
Where
GitHub PR.

Your job shifts from reconstructing behaviour to verifying claims. For each ticked AC, the PR body cites the tagged test and how it was checked in the running app. The spec-traceability gate has already proven every AC ID is covered by a test; the AI review has already hunted for vacuous tests. You confirm the evidence is real and the code is sound.

Merge when satisfied. If the agent amended the spec mid-flight, it'll be called out in the PR description for explicit re-approval — check that.

✓Jira closes the ticket automatically›

On merge of the impl PR, Jira automation rule C transitions the ticket to Done. No manual status update. (Contrast the "old way" tab in the demo, where you had to walk back to Jira and move it by hand.)

✓
That's the loop. Two merges, both by a human, everything else automated. You spent your attention on the two decisions that actually needed judgment: is this the right thing to build, and did it actually get built right.

5 · Ticket types & definition of done

The agent decides what artifact a ticket needs from its type. Know these three so you know what to expect back.

STORY

Needs a spec. EARS acceptance criteria with stable AC IDs at specs/<KEY>/spec.md. Never implemented without an approved (merged) spec. Every AC → a tagged test.

BUG

No spec. A failing repro test named with the ticket key, committed before the fix, is the spec. PR body carries a root-cause note (what was wrong, why this fix, blast radius). Gets the bug label.

TASK

"No observable behaviour change" is the contract. Trivial → no-spec label. Large refactor → an ADR + characterization tests.

◆What counts as "Trivial" (all must hold)›
  • No logic or behaviour change beyond copy, style, or config values.
  • No schema, API, or dependency change.
  • Diff under ~30 lines.
  • No new domain terms.

Claim it with the no-spec PR label; the reviewer verifies the claim. Genuinely unsure? Write the spec — that preserves the human approval gate.

◆The EARS spec format (what a good spec looks like)›

Every acceptance criterion is one EARS statement, testable as written, with a permanent ID (never renumbered or reused once approved — append only):

specs/<KEY>/spec.md
- **AC-1**: When <event>, the system shall <observable response>.
- **AC-2**: If <unwanted condition>, then the system shall <safe response>.
- **AC-3**: While <state>, the system shall <continuous behavior>.
- **AC-4**: The system shall <ubiquitous requirement>.
- **AC-5**: Where <feature flag>, the system shall <conditional behavior>.

The spec also has Goal (one sentence), Out of scope ("spec the negative space"), Open questions, and References (CONTEXT.md terms, ADRs, Confluence).

Do the work

6 · Specs & projects — what you're expected to produce

For a single ticket, the agent drafts the spec. For a project (a feature, an epic, a greenfield slice) you drive the shape yourself, then let the agent fill it in. The tool for both is the same: grill the plan until it's sharp, capture it as a spec, slice it into tickets.

Run inside Claude Code. It reads the ticket via the Atlassian MCP, clones the spec template to specs/<KEY>/spec.md, reformats each acceptance criterion into an EARS statement with a stable AC ID, flags any domain term missing from CONTEXT.md, and comments the spec path + AC list back to the Jira ticket. Then it stops — it does not implement.

inside claude

/grill-me — stress-test the spec before you commit›

This is the single highest-leverage habit. Instead of writing a plan and hoping, you ask the agent to interrogate you relentlessly — resolving each branch of the decision tree until you reach shared understanding. It surfaces the edge cases, the ambiguous ACs, and the "what happens if…" you'd otherwise discover in code review (or production).

inside claude

Use /grill-with-docs when you also want it to sharpen terminology against CONTEXT.md and update the domain docs / ADRs inline as decisions crystallise. That's what keeps the glossary alive.

✓
This is how you modify a spec too. Changing an AC? Re-grill it, update specs/<KEY>/spec.md, and if the change lands mid-implementation, call it out in the impl PR for re-approval (and comment on Jira if the AC itself changed, so the PM sees it).
◆Starting a whole project (not just one ticket)›

For a feature or epic, you are expected to come up with the destination before any agent runs:

  • The PRD / goal — what users need and why. (/to-prd turns a conversation into a PRD on the tracker.)
  • The vertical slices — /to-issues breaks the plan into independently-grabbable tickets, each a thin end-to-end slice ("tracer bullet"), not a horizontal layer.
  • Testable ACs per slice — so each ticket is dispatchable on its own.

Then each slice becomes a normal ticket and flows through §4. The project's coherence lives in the shared CONTEXT.md and any ADRs, not in a giant single PR.

7 · Working a ticket by hand (local agent)

Dispatching a cloud agent is one path. The other is doing it locally with Claude Code — for smaller changes, when you want tight control, or when you're pairing with the agent. Same artifacts, same gates, same PR titles. The only difference is you're driving.

1Pull the ticket & draft the spec›

From inside the repo: claude → /sync-jira EN-1234. Review the draft, then /grill-me to harden it.

Branch hygiene for humans/IDE agents: <KEY>/spec then <KEY>/<short-slug>. (Branch names are for you — automation ignores them.)

2Open the spec PR & get it merged›

Open a PR titled <KEY> SPEC: …, ready for review — never a draft. Link the Jira ticket. Get it reviewed and merged — that's the green light, and the merge pings the control plane exactly as a cloud-agent spec would.

gh
gh pr create --title "EN-1234 SPEC: portfolio CSV export" \
  --body "Spec for EN-1234. See specs/EN-1234/spec.md" --fill
3Implement test-first with /tdd›

One failing test per AC, tagged with the AC ID, before implementation. Then red → green → refactor. Boot the app and acceptance-test the change end-to-end — you'll write that up in the PR body.

4Open the impl PR with acceptance evidence›

Title it <KEY>: … (one ticket = one impl PR). Fill the ## Acceptance evidence section from the template — per-AC table + run output. No evidence files in the diff. The spec-traceability gate will confirm coverage; a reviewer merges; Jira closes the ticket.

The two integrations you lean on locally

Atlassian MCP

Read tickets/Confluence and comment back, from inside the agent. Powers /sync-jira and keeps Jira the PM's source of truth without copy-paste.

GitHub CLI

gh pr create, gh pr view --web, gh pr merge, gh run watch. The agent shells out to it; you use it to review and merge.

⛔
Hard prohibitions (same for humans and agents): don't edit dist/, build/, or generated *.gen.ts; don't commit secrets or force-push main/release/*; don't change package.json dependencies without a specs/<KEY>/ reason. Stop and escalate if two ACs conflict, an ADR would need reversing, or the change touches auth, billing, or PII.

✓ The verification stack

Why you can trust a merge you didn't write line-by-line. Three layers, each catching what the last can't.

1The build proves existence›

spec-traceability blocks the PR if the type-appropriate artifact is missing: an approved spec with every AC ID present in a tagged test (stories), a ticket-key-tagged repro test (bugs), or a no-spec label (trivial). It's a script, not a judgment call, and it reads the ticket key from the PR title. This can't be argued with.

2The agent proves behaviour›

The implementation agent must boot the real app in its sandbox and exercise the change end-to-end before opening the PR. Unit tests passing is necessary, not sufficient. The result is written into the PR body's ## Acceptance evidence — never as committed files.

3The human confirms on evidence›

Each ticked AC in the PR body cites its tagged test and how it was verified in the running app. Reviewers verify claims instead of reconstructing them from scratch. On top of the deterministic gate, the AI review judges whether each tagged test genuinely asserts its criterion and adversarially hunts for inputs that pass the tests but violate an AC — so vacuous tests get blocked.

? "I'm confused" — the fallback

If you take one thing from the presentation and get stuck, start here. These are the questions people actually ask.

?Where do I even start with a new ticket?›

Is it groomed (§4 step 1)? Then either Dispatch Agent in Jira (§4 step 2) and wait for a spec PR, or do it locally: claude → /sync-jira <KEY> → /grill-me (§7). Either way the next thing you'll touch is a spec PR to review.

?The agent opened a PR — do I merge it or does it merge itself?›

You merge, always. A human merges the spec PR (that's approval gate #1) and a human merges the impl PR (gate #2). Agents never merge. Everything between the two merges is automated.

?Why did nothing happen after I merged the spec?›

Most likely the ticket spans multiple repos and the spec isn't yet on the default branch of all of them — the completeness check only trips on the last merge. Otherwise: check the ping-implement-on-spec-merge run in your repo and the implement-orchestrate run in radix-agent-control. To recover manually, run implement-orchestrate.yml via workflow_dispatch with the ticket key.

?The spec PR looks wrong — can I just edit the code and skip the spec?›

No. Fix the spec (re-grill it, edit specs/<KEY>/spec.md), then merge. The spec is the approval gate and the source of AC IDs the CI gate checks against. Skipping it means spec-traceability blocks the impl PR anyway.

?CI is stuck on a check that never reports.›

A required check got skipped by a paths-ignore filter (common on markdown-only spec PRs) and so never reports its status. Required checks must "always run and self-skip" like spec-traceability does. This is a repo-owner fix — see §3's CI-hygiene note. Don't disable branch protection to get around it.

?Cursor launch failed with "Failed to verify existence of branch."›

Misleading error. It almost always means the CURSOR_API_KEY in the control plane isn't a User/Service Account key, or its Cursor GitHub connection doesn't include the target repo. Team keys are rejected. This lives in radix-agent-control — flag it to whoever owns the control plane.

?Where do questions go?›
  • Product question → comment on the Jira ticket; the PM answers there.
  • Engineering question → Open questions in the spec; the developer resolves.
  • Minor ambiguity → log an explicit assumption in the spec; it gets reviewed at spec approval.
?Can I put a screenshot in the PR to prove it works?›

No — no evidence files in GitHub, anywhere. The PR diff is code and tests only. Put the acceptance account in the PR body's ## Acceptance evidence section (per-AC table + run output); visuals stay in the linked agent run.

?Who do I ask when this doc doesn't cover it?›

The kit's own docs are the deeper reference: docs/agentic-dispatch.md, docs/jira-setup.md, docs/agent-ready-checklist.md, docs/metrics.md, and the ADRs in docs/adr/. The control plane's flow is documented in radix-agent-control/README.md.
